# life-plan-lab

FIRE・住宅ローン・教育費・老後を統合した生活設計シミュレーター（V2）。

「未来を当てる」ためではなく「未来を整理する」ためのツール。入力して終わりではなく、
**結果を見る → 条件を変える → また結果を見る → 家族で話す** という試行錯誤を支えることを重視しています。

## 開発

```bash
npm install
npm run dev      # 開発サーバ（http://localhost:5173）
npm run build    # 型チェック + 本番ビルド（dist/）
npm test         # ロジック/ストア/描画のテスト（vitest）
```

静的アプリとしてビルドし、将来 `/lifeplan-v2/` などのサブディレクトリに独立ページとして
公開する想定（`vite.config.ts` の `base: './'` で相対パス出力）。

## アーキテクチャ

- `src/schema` … 型・入力スキーマ（`Field<T>{value, source, label, assumptionText}`）・標準値/おすすめ値・正規化（両モードの合流点 `buildFullInput`）・ざっくり診断の質問定義
- `src/engine` … 純粋関数の計算群（年次/教育費/住宅ローン/FIRE/判定）。UIから完全分離。定数は `engine/constants.ts`
- `src/store` … zustand。ステップ遷移・入力状態・自動保存（localStorage）
- `src/features` … 画面（mode-select / input-steps / results / resume）

## 計算の反映ステータス（STEP5）

コード上は `src/engine/annualSimulationEngine.ts` 冒頭のコメントに、表示上は結果画面の「今回の試算条件」注記に反映状況を記載しています。

**反映済み**：年齢／**手取り年収（直接入力＞夫婦別収入の手取り換算合算＞世帯年収×年収帯手取り率）**／
**昇給率（現役期に年次反映）**／退職予定年齢（以降は労働収入0）／現在資産（**現金/投資に分割**）／
**現金比率（投資資産のみ利回り）**／**毎月投資額（現金→投資への振替。収支に二重加算しない）**／毎月生活費／
住宅費（賃貸=家賃・持ち家=ローン返済を完済年齢まで＋**完済後維持費 年60万**）／子ども別年齢→教育費／教育方針／
想定利回り（名目）／インフレ率（支出側）／FIRE種別・希望年齢／FIRE後生活費／サイドFIRE後収入・就労終了年齢／
年間特別費・車・旅行・保険／退職金（**FIRE/退職年に一括**）／年金（65歳〜・入力時）／医療介護予備費（75/85歳〜）／
ライフイベント支出・収入（相続は収入・断定しない表現）

**簡略反映**：持ち家維持費は定額（年60万）／教育費は初期値テーブル／税（0）／手取り率は年収帯ざっくり

**未反映（次STEP）**：住宅ローン残高・金利・固定変動・返済方式・ボーナス払いからの償却／配偶者年齢／
NISA・iDeCo・各種控除／所得税・住民税・社会保険料の精密計算／暴落シナリオ／夫婦別の退職年齢・年金

**資産モデル（重要）**：資産を現金資産と投資資産に分け、利回りは投資資産にのみ適用。年間収支は現金へ加え、
現役期かつ黒字のときのみ毎月投資額（未入力時は黒字の一部）を現金→投資へ移動（赤字年は新規投資を停止）。
赤字は現金→投資の順で取り崩し。毎月投資額は「新規投入額」であり収支に二重加算しません。

**運用順序（保守的）**：利回りは「年初の投資資産」にのみ適用し、その年の新規投入は翌年から運用します。

**枯渇の扱い（STEP5.1で修正）**：現金・投資資産はマイナスにしません。取り崩せない不足分は
`cumulativeShortfall`（累計不足額）に蓄積し、**マイナス残高に負の利回りをかけない**（負の複利を発生させない）。
表示資産は0でクランプし、結果には「◯歳で資産が尽きる見込み・95歳時点の累計不足額：約◯◯万円」と表示します。

**FIRE準備率（4%ルール）は参考指標**：Hero では資産寿命・95歳残資産を主に見せ、FIRE準備率は目安として併記。
年金未入力時はその旨を注記し、年次シミュレーションの資産寿命とあわせて確認するよう案内します。

> ざっくり診断にも、現実感のための最小項目（毎月生活費・毎月住居費・住宅ローン残年数・子の年齢・FIRE後生活費・サイドFIRE後収入）を追加しました。
> これらは同じ意味のしっかり診断項目へ引き継がれ、別管理になりません（buildFullInput で合流）。

## 入力の重複防止（STEP5.2）

同じ支出を2回入力させない／計算で二重計上しないためのルール：

- **質問順**：毎月固定費（生活費・保険料）→ 毎年変動費（特別費・旅行・車維持費）→ 一時イベント（車購入・リフォーム・相続など）。
- **役割分担**：年間特別費は「他で個別入力しない不定期支出」。旅行費・車維持費・車購入を別入力する場合は特別費から除く（Help に「含めない／入力不要」を明記）。
- **車**：車関連費＝毎年の維持費、車購入＝ライフイベントの一時支出（別物）。
- **生活費**：毎月生活費は住居費・教育費・投資額・保険料を含めない。
- **住居費**：持ち家でローン返済額があればそれを優先（家賃は賃貸のみ）。ボーナス払いは毎月返済額に均して入力（ボーナス払い欄は記録用）。
- **FIRE後／老後生活費**：FIRE開始〜65歳前は FIRE後生活費、65歳以降は老後生活費（年齢で切替・合算しない）。
- **教育費**：子ども別設定のみで算出（教育方針は初期値の補完のみ・二重加算しない）。
- **退職金**：入力は1か所（老後ステップ）。退職／FIRE開始の年に一時収入として反映。

## 表示の整合性（STEP5.2）

Hero・今回のポイント・タイムライン・資産推移グラフ・年次テーブルは、すべて同一の `runSimulation` 結果と
`buildLifeEvents`（イベント単一ソース）を参照します。資産枯渇年齢は1つに統一され、95歳時点はマイナスではなく
0＋累計不足額で表示します。FIRE準備率（4%ルール）は参考指標で、**資産寿命が短い場合は総合判定を「安定」にしません**
（年次シミュレーションの資産寿命を優先）。

## 現在価値換算（STEP5.2 追加）

計算エンジンは名目額（将来額）ベースのままですが、表示用に現在価値を併記します。

- **換算式**：現在価値 = 将来額 ÷ (1 + インフレ率)^(対象年齢 − 現在年齢)。各 `YearRow.debug` に `presentValueFactor` と
  各値のPVを保持。
- **累計不足額**は各年の不足額を**その年の係数で割り戻して合計**（`cumulativeShortfallPresentValue`）。
- **表示は現在価値を主役**：Hero/今回のポイントは現在価値を主に、将来額（名目）は補足。資産推移グラフの詳細表は
  「資産額（将来）」と「現在価値」を併記。グラフ本体の現在価値切替は今後対応（TODO）。
- 注記：「将来額はインフレを反映した金額です。現在価値は、今のお金の感覚に割り戻した目安です。」
- **実質利回りの目安**：名目利回りと支出インフレから実質利回り＝(1+名目)/(1+インフレ)−1 を注記（例：5%/2%→約2.9%）。
- **枯渇時のHero表示**：「95歳時点 0万円」ではなく「資産寿命：◯歳ごろ／95歳時点：資産は枯渇済み／累計不足額（現在価値・将来額）」を表示。
- **年金未入力の影響を明示**：未入力時は「65歳以降の収入を0円として計算。年金を入力すると資産寿命が大きく変わる可能性」と案内。
- 開発用サンプルは「高収入（年金未入力・検証用）」と「高収入（年金あり・現実寄り, 年金330万）」に分離。

## 計算反映・再計算の信頼性（STEP5.3）

- **入力反映の総点検**：全カテゴリで「結果→修正→再計算」が結果に反映されることをテストで担保（感度テスト：収入/生活費/利回り/インフレ/年金/老後生活費/昇給/退職年齢/現金比率/ライフイベント/相続）。
- **枯渇後の黒字＝累計不足額の返済（§3）**：一度枯渇した後に年金等で黒字化したら、まず `cumulativeShortfall` を返済し（穴を埋め）、0に戻ったら資産を再形成。`assetLongevityAge` は「初めて尽きた年齢」として履歴に残す。これにより老後入力の変更が累計不足額に反映される。
- **大学区分の細分化**：なし／国公立文系／国公立理系／私立文系／私立理系／未定（住まいは自宅／一人暮らし／未定）。未定は標準的な仮定（国公立文系・自宅）。
- **入力UI**：金利などの小数入力に対応（`NumberField`：`inputMode="decimal"`、途中の "0." を保持、Backspaceで空欄に戻せる・デフォルト再注入なし）。子ども年齢が未入力のときは「年齢未入力」と表示（仮値を確定表示しない）。
- **枯渇時の表示統一**：Hero・資産推移カードとも「◯歳ごろ枯渇／95歳時点は枯渇済み／累計不足額（現在価値）」に統一（「95歳時点 0万円」を主表示にしない）。
- `SimulationResult.calculatedAt` を追加（再計算が走っているかの確認用）。

## 計算エンジンの検証強化（STEP5.4）

- **貸借一致（reconciliation）**：各年 `当年末資産 = 前年末 + 運用益 + 年間収支 − 不足返済 + 不足追加`。
  `YearRow.debug.reconciliationDiff` が常に ≈0（テストで全年検証）。
- **手計算ゴールデンケース**：利回りのみ／現金のみ／毎年赤字／インフレのみ等を、エンジンから導出しない固定期待値で検証
  （`tests/engine/golden.test.ts`）。独立式（複利）とも照合。
- **詳細グラフは2本線**：Bottom Sheet の拡大グラフで「将来額（補助線）」と「現在価値（主線）」を凡例つきで比較。
  通常カードのミニグラフは現在価値1本。年次表は将来額と現在価値を併記。
- **単位**：月額入力（生活費・住居費・FIRE後生活費・サイドFIRE収入）は ×12 で年額化、年金/老後生活費/退職金は年額・一時金、
  金利/利回り/インフレは%→小数。テストで固定。

### 入力項目の反映ステータス

## 結果が厳しすぎる原因の特定と前提整理（STEP5.5）

実機の高収入ケースが「64歳枯渇・95歳累計不足額 将来額2.3億」と過度に厳しく出た原因を年次内訳で切り分け、修正。

- **主因＝収入だけ名目固定・支出だけインフレ**：年金240/サイド400が固定のまま支出は95歳で1,250へ膨張していた。
  → **FIRE後収入・年金・退職金・相続を「現在価値入力」とみなし、支出と同じインフレ率で将来額へ換算**（収入と支出の前提を統一）。
  この1点で 枯渇 64→75歳、累計不足額 将来額2.3億→8,777万・現在価値9,618→3,420万 に改善。
- **FIRE後/老後生活費は「日常生活費のみ」**と明示（住居費・教育費・保険・特別費・旅行・車関連費は別加算）。Help と注記に記載。
- **特別費・旅行費・車関連費は老後も毎年継続**する前提を注記（老後に減らすと改善する旨）。
- **暴落シナリオは現時点で記録用（未反映）** を注記（結果悪化は暴落のせいではないと明示）。
- **毎月投資額は黒字の範囲でのみ現金→投資へ振替**（黒字超過分は投資されない・総資産を直接増やさない）を注記。
- **現在資産の表示ズレを修正**：年次表・グラフの先頭を「◯歳（現在）＝入力した現在資産（年初）」とし、以降を年末資産に統一。
- **見直しが効きやすいポイント**（最大5件）を結果画面に表示：枯渇年齢・教育費ピーク・FIRE後までのローン・年金<老後支出・老後の裁量支出継続・年金未入力 など。
- 比較シナリオ（特別費/旅行/車を0・年金増・子なし・一時イベントなし・老後生活費減）で、各レバーが資産寿命を改善することをテストで確認。

## 入力反映の検証と修正導線の粒度（STEP5.6）

「月20万円投資しているのに資産が伸びにくい」という実機所感を、年次内訳で検証して原因を明示。
新機能追加ではなく、反映状況の確認・説明・修正導線の改善が目的。

- **毎月投資額は家計の黒字に上限**であることを年次内訳で可視化。検証ケース（手取り650・月20万投資）では
  計画240万/年に対し、実際の積立は初年度28万・43歳以降は赤字のため0万。資産が伸びにくいのは
  「満額を積み立てられていない」ためで、計算は正しい（既存資産の運用益のみで微増）。
  - 年次デバッグに `plannedInvestmentAmount`（満額の意図）/ `actualInvestmentAmount`（実際）/ `skippedInvestmentAmount`（積み立てられなかった分）を追加。
  - 指標に `monthlyInvestmentPlannedAnnual`・`investmentUnderfundedFromAge` を追加し、満額に届かない最初の年齢を把握。
  - **見直しが効きやすいポイント**に「毎月投資額が黒字を上回り◯歳ごろから満額積立できていない（生活費・住居費を見直すと余力が増える）」を追加。
- **一時イベント（車購入・リフォーム等）を節目の単一ソースに追加**：計算（一時支出）と同じ入力から、グラフのマーカー・年次テーブルにも反映。
  年次テーブルは「5年ごと＋イベント年」を表示。
- **変動金利のとき「固定終了年齢」を非表示**にし、タイムラインの固定金利終了マーカーも出さない（住宅費計算には元々未反映＝記録用）。
- **しっかり診断の結果では「もっと正確に見る／しっかり診断で詳しく見る」を非表示**（既に詳細なため）。ざっくり診断の結果では従来どおり表示。
- **結果画面の修正導線をページ単位に拡張**：表示中のしっかり診断ページ（年金だけ・投資額だけ・車購入だけ等）へ直接戻れる。
  入力値は保持され、`再計算して結果へ`で結果に戻る。
- **住宅ローン完済（現在年齢＋残年数）後は維持費のみ**で、返済額との二重計上がないことをテストで確認（38+30=68歳で完済→以降は年60万）。

## 結果画面の情報量調整＋実際に反映された積立額の表示（STEP5.7）

「正しいけれど、表示されないと納得できない」を解消するための調整。計算ロジックは変更なし。

- **実際に計算へ反映された積立額を明示**：指標に `monthlyInvestmentActualFirstYear`（初年度の反映額）・`monthlyInvestmentActualAverage`（現役期の平均反映額）を追加。
  「見直しが効きやすいポイント」の該当項目に、入力額と実際の反映額を併記。
  例）検証ケースでは「入力上 月20万円（年240万円）→ 実際に反映 初年度 年28万円（月約2.3万円）、現役期平均 年4万円ほど。38歳ごろから満額は積み立てられていません」。
- **結果画面の情報量を整理**（スマホのスクロール量を抑制）。
  - 常時表示：総合結果・資産寿命・95歳時点・今回のポイント・人生の主な節目・資産推移/教育費/住宅ローンの各カード。
  - 折りたたみ（初期は閉じる・矢印で開閉）：見直しが効きやすいポイント（◯件）・条件を変えてみる・見直しのヒント・今回の試算条件。
- 折りたたんでも修正導線・再計算は維持（ページ単位の修正ボタン→該当ステップ遷移→`再計算して結果へ`）。ざっくり/しっかり両モードで表示が破綻しないことをテストで確認。

## 世界観・心理UX・見やすさ改善（STEP6）

「FIRE計算機」ではなく「人生の見通しを整理するダッシュボード」へ。計算ロジックは変更なし、UI/UX・文言・構成のみ。
思想：見通し・終わりが見える・原因が分かる・やり直せる、を静かに伝える（煽らない／危機感で動かさない）。

- **人生フェーズ＋次の節目カード（`Outlook`）** を結論(Hero)の直後に常時表示。
  - 現在のフェーズ（資産形成期／教育費準備期／教育費ピーク期／FIRE準備期／サイドFIRE期／FIRE期／老後移行期／年金生活期）を年齢・FIRE・年金の位置から判定（`lifePhase.ts`）。
  - 「次に確認したい節目」：現在より先の前向きな節目を最大3件（枯渇・現在・95歳は除外）。静かな表現に統一。
  - 「この先の見通し」：教育費ピークの一段落・住宅ローン完済（FIRE後に残るか）など“終わりが見える”情報を添える。
- **見直しが効きやすいポイントを構造化**：各項目を「見出し＋原因／時期／どこを変えると改善」の短い箇条書きに分解（`RiskFactor{title, points[]}`）。
  「悪い結果」ではなく「どこを調整すればよいか」に視点を移す。
- **自分専用感の文言**：結果見出しを「あなたの人生ダッシュボード」に。家族名は入力がない限り表示しない。
- **控えめなモーション**：結果カードの一度きりのフェード、折りたたみ展開時のフェード、ボタン押下の微反応。`prefers-reduced-motion` で無効化。
- **構成（常時表示／折りたたみ）は STEP5.7 の思想を維持**：結論・今回のポイント・主な節目・資産推移/教育費/住宅ローンは常時表示、見直しポイント・条件変更・試算条件・注記は折りたたみ。
- 入力画面はスコープ外（大改修なし）。ステップ遷移時の先頭スクロールは既存実装で維持。

### WordPress 固定ページへの iframe 埋め込み

```html
<iframe

```

### WordPress テーマ（Lightning 等）との干渉メモ

固定ページの本文幅が狭いと窮屈になるため、必要に応じて固定ページ限定でCSS（カスタムHTML内 `<style>` か追加CSS）を：

```html
<style>
  /* iframe を本文幅いっぱい〜画面いっぱいに。固定ページ限定で適用すること。 */
  .lifeplan-embed { width: 100%; }
  .lifeplan-embed iframe { width: 100%; border: 0; display: block; }
  /* 本文の左右パディングが強い場合のフルブリード（テーマにより調整） */
  @media (max-width: 782px) {
    .lifeplan-embed { margin-left: calc(50% - 50vw); margin-right: calc(50% - 50vw); }
  }
</style>
<div class="lifeplan-embed">
  <iframe src="（公開URL）" title="生活設計シミュレーター" style="min-height:1000px;" loading="lazy"></iframe>
</div>
```

- iframe 既定の枠線・余白は `border:0; display:block` で消える。
- 背景はウォームアイボリー固定。テーマが白基調なら自然に馴染む。
- 下部固定ナビ・グラフ拡大 Bottom Sheet は iframe ビューポート基準で完結（親ページには影響しない）。

### iframe 高さの方針

- まずは `min-height: 1000px` 固定で運用（結果画面まで概ね収まる目安）。
- 余白が気になる/結果が長い場合は、将来 postMessage による auto-resize を検討（STEP9で候補化）。最初から入れる必要はない。

## UI質感調整（STEP10.1）

「機能的で整理されたUI」を保ったまま、静けさ・読みやすさを一段上げる調整（ロジック・導線・入力・iframe構成は不変）。

### ① カード境界を弱める（箱感→面の呼吸感）

- `--border` を `#e4e0d6 → #ebe7dd`（カード色に近づけ、線を弱める）。
- `--shadow` を常時ほぼ無し `0 1px 2px rgba(35,50,42,.03)` に（大きなアンビエント影を撤去）。浮きは `--shadow-raised` をホバー時のみ（mode-card）。
- Hero / Outlook は常時 raised をやめ通常影＋淡い緑グラデで「面」で区切る。details（試算条件・慎重条件・見直し）も同じ `--shadow` トーンに統一。
- 結果セクション間 `.result { gap: 18px }`（カード内 padding は維持）。Hero の padding を 22→24px に。

### ② Hero の情報密度を下げる（分析画面→人生の要約）

旧：資産寿命／95歳時点／FIRE準備率 を3指標グリッドで横並び（KPIダッシュボード感）。
新：**主役＝資産寿命**（1.7rem）、**支え＝95歳時点（現在価値＋将来額）**、**脚注＝FIRE準備率（目安・muted）**。判定バッジは維持。教育費ピーク・ローン完済・FIRE後の状態は直下の Outlook カードが担うため Hero では重複させない。

| 情報 | 旧 | 新 |
|---|---|---|
| 資産寿命 | 3指標の1つ | 主役（大きく） |
| 95歳時点(PV) | 3指標の1つ | 支え（中・muted） |
| FIRE準備率 | 3指標の1つ | 脚注（小・muted・「目安」） |
| 内部スコア(/15) | バッジ横 | 維持（小） |

### ③ グラフを主役寄りに

- 通常表示グラフ高さ 120→**150px**、カード内で左右にやや広く（`margin: 10px -8px; width: calc(100% + 16px)`）。
- 拡大グラフ高さ 280→**320px**、余白増。

## 空気感の最終調整（STEP10.2）

## 世界観の最終微調整（STEP10.3）

「綺麗なFPツール」から「未来を静かに眺める人生設計UI」へ寄せる。構造・計算・導線は不変、視線・空気感・情報階層・文言のみ。
